# 第十阶段：性能压测与容量基线手册

## 1. 目标

性能治理覆盖功能房查询、预约冲突检查与创建、签到状态、审批队列、通知读取和管理报表。测试必须在隔离数据库执行，不得把写入型压测直接指向生产环境。

基础验收门槛：

- HTTP 请求错误率不高于 1%；
- 常用接口 P95 不高于 800 ms，P99 不高于 1500 ms；
- CI 小规模容量烟雾测试 P95 不高于 1500 ms；
- 代表性数据库查询 P95 不高于 500 ms；
- 数据库连接池等待队列长期保持为 0，短时不超过 5；
- 慢查询比例低于 10%；
- 预约并发测试不得出现重复占用；
- 通知 Outbox 不得因压测持续积压或产生异常死信。

上述数值是首版发布门槛，不代表服务器的永久容量。更换服务器、数据库规格、网络拓扑或数据规模后必须重新测量。

## 2. 压测模型

### 2.1 正常负载

- 目标吞吐：约 20 请求/秒；
- 并发用户：约 30；
- 持续时间：10 分钟；
- 主要流量：房间列表、时间轴、个人预约和通知查询。

### 2.2 预约开放高峰

- 目标吞吐：约 80 请求/秒；
- 并发用户：约 120；
- 持续时间：10 分钟；
- 流量组成：房间查询 25%、时间轴 20%、冲突检查 20%、预约提交 10%、个人预约列表 10%、签到与通知 10%、管理端查询 5%。

预约提交必须为每个虚拟用户生成独立幂等键，并使用隔离数据集中的不同座位或时间段。409 冲突是业务结果，不计作服务器错误；同一座位和时间段成功超过一次则判定测试失败。

### 2.3 签到与审批高峰

- 目标吞吐：约 30 请求/秒；
- 并发用户：约 60；
- 持续时间：5 分钟；
- 同时执行签到状态查询、工作人员签到、待审批列表、审批更新和通知读取。

写入型签到与审批必须预生成测试预约。重复签到或重复审批返回 409 属于预期竞争结果，5xx、审计缺失或通知 Outbox 丢失属于失败。

### 2.4 报表负载

- 目标吞吐：约 10 请求/秒；
- 并发管理员：10；
- 持续时间：10 分钟；
- 覆盖仪表盘、预约统计、使用率、高峰时段、爽约统计和导出前置查询。

报表压测应使用接近生产规模的数据副本，避免只在十几条种子数据上得出错误结论。

### 2.5 压力与极限测试

从 20 请求/秒开始，每 3 分钟增加 20 请求/秒，直到出现下列任一情况：

- HTTP P95 连续两个窗口超过 1500 ms；
- 5xx 超过 1%；
- 数据库连接池队列持续增加；
- Redis 或 MySQL CPU、内存、连接数达到安全上限；
- 通知 Outbox 延迟持续增加。

达到停止条件后立即降载，记录系统恢复时间。极限值只用于容量规划，不得直接作为生产日常目标。

## 3. 自动化检查

### 3.1 独立遥测回归

```bash
npm run check:performance
```

检查数据库查询计数、错误、慢查询、耗时直方图、连接池告警和性能索引启动门禁。

### 3.2 数据库索引与执行计划

```bash
MYSQL_HOST=127.0.0.1 \
MYSQL_USER=root \
MYSQL_PASSWORD=<密码> \
MYSQL_DATABASE=<隔离数据库> \
npm run check:performance:mysql
```

该检查验证必要复合索引、代表性查询执行计划和数据库 P95。报告可通过 `PERF_DB_REPORT_FILE` 输出为 JSON。

### 3.3 HTTP 容量烟雾测试

```bash
PERF_BASE_URL=http://127.0.0.1:3000/api/v1 \
PERF_STUDENT_TOKEN=<测试学生访问令牌> \
PERF_ADMIN_TOKEN=<测试超级管理员访问令牌> \
PERF_HTTP_REPORT_FILE=performance-http-report.json \
npm run check:performance:http
```

该脚本为有界并发烟雾测试，覆盖公共房间查询、学生预约与签到状态、通知读取以及管理员审批和报表读取。它不是生产极限压测工具，不会执行预约、签到或审批写入。

GitHub Actions 的 `Performance and capacity checks` 工作流会在真实 MySQL、Redis 和启动后的 API 上运行上述检查，并保存数据库、HTTP 和服务日志作为验收证据。

## 4. 数据库容量配置

生产默认建议：

```env
MYSQL_CONNECTION_LIMIT=20
MYSQL_MAX_IDLE=20
MYSQL_IDLE_TIMEOUT_MS=60000
MYSQL_QUEUE_LIMIT=1000
MYSQL_CONNECT_TIMEOUT_MS=10000
DB_SLOW_QUERY_MS=250
```

连接池不能只通过增加连接数解决排队问题。调整前应同时检查：

1. 慢查询和执行计划；
2. API 实例数；
3. MySQL `max_connections`；
4. 每个实例连接池上限乘以实例数后的总连接数；
5. 事务持有时间和外部网络调用是否发生在事务内。

建议总连接预算不超过 MySQL 可用连接数的 70%，保留管理、迁移、备份和故障处置余量。

## 5. 新增性能索引

- 房间：状态、类型、楼栋；
- 座位：房间、状态和展示顺序；
- 预约：审批状态与创建时间；
- 预约：用户、状态和日程；
- 预约：房间、日期、状态、座位和时间；
- 签到：房间和签到时间；
- 通知：用户、已读状态和创建时间；
- 候补：状态和创建时间。

迁移命令：

```bash
node scripts/apply-performance-indexes-migration.js
```

迁移使用 MySQL 命名锁并可重复执行。API、Scheduler Worker 和生产 Readiness 均会拒绝在索引不完整时启动或通过。

## 6. 监控指标

新增指标：

- `jingyi_database_queries_total`；
- `jingyi_database_query_errors_total`；
- `jingyi_database_slow_queries_total`；
- `jingyi_database_query_duration_seconds`；
- `jingyi_database_pool_connections`；
- `jingyi_database_pool_free_connections`；
- `jingyi_database_pool_queued_requests`。

新增告警：

- `DATABASE_ERROR_RATE`；
- `DATABASE_SLOW_RATE`；
- `DATABASE_POOL_QUEUE`。

慢查询日志只记录 SQL 指纹、操作类型、耗时和错误码，不记录参数或用户数据。

## 7. 调优顺序

1. 确认错误不是业务冲突、限流或错误测试数据；
2. 查看 HTTP 路由、数据库耗时和连接池队列；
3. 检查慢查询指纹和 `EXPLAIN FORMAT=JSON`；
4. 优化 N+1 查询、分页、过滤条件与索引；
5. 再评估连接池和实例数量；
6. 完整重跑正常、峰值和恢复测试；
7. 将新基线、服务器规格、数据规模和提交号写入验收记录。

## 8. 发布容量判定

只有在以下条件同时满足时，才能判定性能验收通过：

- 性能工作流全部通过；
- 测试报告和服务日志已保存；
- 预约并发一致性回归通过；
- 关键查询执行计划使用预期索引；
- 观察期间没有数据库、Redis、Outbox或审计告警；
- 实际生产规格的峰值测试至少保留 30% 容量余量。
