# 第四阶段：运行时协调与实时通道安全

本阶段在数据库一致性改造完成后，处理多实例部署下的定时任务重复执行、WebSocket 未鉴权、运行时事件广播和部署拓扑问题。

## 目标

1. 定时任务在多进程或多实例部署时只由一个执行者运行。
2. 每个任务具有 Redis 分布式锁、唯一执行标识、超时与安全释放机制。
3. Socket.IO 连接必须携带并验证访问令牌。
4. 客户端只能加入经过授权的房间，禁止任意字符串房间加入。
5. 为后续 Socket.IO Redis Adapter 和水平扩展预留统一接口。
6. 增加运行时协调检查、回归测试和部署说明。

## 第一批改造

- 新建运行时分布式锁服务；
- 重构 schedulerService，所有任务通过统一锁包装器执行；
- 防止同一进程重复初始化定时任务；
- 对任务记录 acquired、skipped、success、failed 和 duration；
- production 启用 scheduler 时强制要求真实 Redis；
- 增加 scheduler 静态和运行时回归检查。

## 第二批改造

- Socket.IO 握手鉴权；
- 访问令牌类型、黑名单和用户状态校验；
- 房间白名单与角色授权；
- 连接频率及事件输入限制；
- 对现有管理端 Monitor 页面补充 token 传递。

## 第三批改造

- 评估并接入 `@socket.io/redis-adapter`；
- 分离 pub/sub Redis 连接；
- 增加多实例广播集成检查；
- 编写 PM2 / 容器部署拓扑说明。

## 非目标

- 本阶段不进行大规模 UI 重构；
- 不修改预约业务规则；
- 不更换数据库；
- 不在没有回归测试的情况下直接启用生产实时广播。
